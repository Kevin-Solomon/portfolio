Order Placement Outage — 14-Hour RCA & Recovery

Led the investigation and resolution of a critical 14-hour outage where 100% of order placements failed. Identified root cause, restored service, and established 6 new engineering practices to protect the order pipeline.
14 hrs
Outage Duration
100%

What Happened

A routine deployment (ShipRocket integration) included a debugging tool called axios-curlirize that had a hidden flaw. It created a circular reference in the data it handled, like two mirrors facing each other creating infinite reflections. When our logging system tried to record this data, it crashed. Since logging runs on every order request, every single order failed from 10:50 PM on May 12 until 12:40 PM on May 13. No automated alert existed for order failures, so the outage went undetected for ~12 hours until the marketing team noticed no orders had come through.

What I Did

Traced the failure to the recent deployment and rolled it back to restore service immediately. Then did a deeper investigation the next day to find the true root cause: it wasn't just the debugging tool, it was that our logging system was fragile enough that a single bad data structure could take down the entire order pipeline. Made the logging system resilient to handle unexpected data without crashing. Prepared a full incident report for leadership with root cause analysis, business impact (251 impacted users, 76 of whom had clicked place order), and a retargeting campaign plan for affected customers.

Impact
  • Service fully restored with zero data loss. No partial or corrupted orders were created
  • Root cause identified beyond the surface, rolling back fixed the symptom, but the investigation revealed the logging fragility that could have been triggered by something else later
  • Logging system hardened so it can never take down a customer request again
  • Retargeting campaign planned for the 251 impacted users (76 had clicked place order) to recover lost conversions
Practices Established
  • Real-time order health monitoring, automated alerts fire within minutes if order success rates drop. No more 12-hour blind spots
  • Mandatory staging validation before production for any change touching order flow, no exceptions
  • Third-party tool vetting for anything that runs alongside the order pipeline, including helper and debugging tools
  • Post-deployment smoke test, an automated test places a sample order end-to-end after every deploy. If it fails, the deployment is flagged immediately
  • Crash-proof supporting systems, logging, analytics, and debugging tools are isolated from order processing. If they fail, they fail silently
  • Quarterly order flow resilience reviews to catch new risks before they become incidents
Bottom line: Turned a critical 14-hour outage into a catalyst for 6 permanent engineering practices that protect the most important pipeline in the business. The fix wasn't just restoring service, it was making sure this class of failure can never happen again.

What Happened

Third-party API failures cascaded through the order pipeline unchecked, causing retry storms that sent 15+ duplicate messages per customer. Each incident created 100+ ghost orders on Shopify that had to be manually cancelled, cross-verified with the internal system, and reconciled against the database. The full cleanup took 1-2 hours every time.

What I Did

Impact
  • $42,000/year in revenue protected — 100 orders at risk per incident, at least once a month
  • 36 engineering hours/year saved — eliminated 1-2 hours of manual Shopify cleanup per incident
  • 18,000+ spam messages/year prevented — stopped 1,500+ unwanted messages per incident from hitting customers
Bottom line: A few try-catch blocks turned $42,000/year in recurring chaos into a non-issue. The hidden cost was customer trust — a single spam storm of 15+ messages per customer drives chargebacks, negative reviews, and permanent churn.

What Happened

Every day we received at least 2-3 queries about Snapmint orders not syncing — on high-traffic days the number would double. Snapmint's APIs were taking too long to resolve order statuses, leaving paid orders unsynced in our database and Shopify. Customers would see no confirmation, panic, and flood CX on WhatsApp demanding cancellations or refunds — sometimes a week after paying. This was destroying trust. Bottom line: A single cron job eliminated a daily source of CX complaints, freed up engineering time that was being lost to repetitive manual fixes, and restored customer trust in the payment experience. The best part — since going live, not a single complaint from CX about unsynced Snapmint orders.

OTP Login Fixes — WhatsApp + SMS Flows

Found and fixed bugs blocking first-time users from WhatsApp OTP signup (1,500 Sentry events) and breaking the SMS fallback flow between Msg91 and Kaleyra.

What Happened

Two separate bugs were breaking our OTP login flows. First, every first-time user trying to sign up via WhatsApp OTP hit a 500 error and saw "An error occurred while sending the OTP." Only the WhatsApp path was broken for new customers, silently blocking new customer acquisition. Second, our SMS OTP flow had a cascading failure: Msg91 (primary SMS provider) would sometimes return 500 errors, which triggered the Kaleyra fallback. But Kaleyra's OTP template map didn't have the current domain name (curaahome.com), only the old one (yflhome.com), so the fallback also crashed. Both SMS providers failing meant some users couldn't log in at all.

Root Cause — WhatsApp OTP

The OTP handler fetches the customer from the database by phone number. For first-time users, no record exists so the lookup returns undefined. A guard condition had a falsy evaluation bug: !customer && required evaluated to true && undefined → undefined (falsy), so the error handler was skipped. The code passed undefined as the customer to the WhatsApp OTP sender, which crashed on customer.phoneNumber.

Root Cause — SMS OTP Fallback

When Msg91 returned a 502, the Kaleyra fallback fired. But the otpTemplates map only had an entry for yflhome.com (the old domain), not curaahome.com (the current domain). So otpTemplates["curaahome.com"] returned undefined, and accessing .content on it threw a TypeError. This meant whenever Msg91 was down, the Kaleyra fallback was also broken, and users got no OTP at all.

What I Did

For the WhatsApp bug: fixed the logic so sending OTP doesn't require a customer record to already exist. If the customer isn't in the DB yet (first-time user), the fix uses the phone number from the request input instead of relying on the customer object. For the SMS bug: added the current domain (curaahome.com) to Kaleyra's OTP template map so the fallback actually works when Msg91 goes down.

Impact
  • All first-time WhatsApp signups unblocked, new customers can now sign up via WhatsApp OTP without hitting a 500 error
  • SMS fallback actually works now, added the curaahome.com domain to Kaleyra's template map so when Msg91 goes down, customers still get their OTP
  • 1,500+ Sentry errors resolved across both bugs, cleaning up some of the highest-volume error sources
  • Both login paths secured, WhatsApp OTP and SMS OTP now work reliably for new and existing customers
Bottom line: Two subtle bugs were silently breaking both OTP login paths. The WhatsApp fix unblocked all new customer signups, and the SMS fix ensured the Kaleyra fallback actually works when Msg91 goes down. Small fixes, critical impact on customer acquisition and login reliability.
